考察
最適化レベルを上げると処理がはやくなった。
また、grayscaleにしても処理が早くなった。
-o1,o2,o3はそれぞれコードを書き換えているけど、その書換の量が数字が増えていくと多くなっていく
それは内部では以下のことを行っているからである(S1)

| オプション    | 主な実装内容                           | 目的     |
| -------- | -------------------------------- | ------ |
| `-O0`    | 最適化なし                            | デバッグ   |
| `-O1`    | 基本的なコード削減、定数畳み込み、不要処理削除          | 軽量高速化  |
| `-O2`    | `-O1` + 関数展開、共通式削除、ループ最適化など      | 実用的高速化 |
| `-O3`    | `-O2` + ループ展開、SIMD化、より積極的なインライン化 | 速度最大化  |
| `-Ofast` | `-O3` + 浮動小数点規則を緩和               | 速度最優先  |

実行時になるべく計算を行わないことや並列実行などで速さを稼いでいる
しかし、関数が複雑になったり浮動小数点演算で誤差が発生する場合は-03や-0fastは思わぬエラーになると考えた。
また、グレースケールにすると処理時間が２倍から３倍ほど短くなっている
これは４重ループ内の一回のループで計算する量がcolorではr,g,bの３回に対して、grayscaleにすると一回になるからだろう
また、キャッシュに乗りやすい側面もあるだろう
SIMD化というものもあるらしい。
